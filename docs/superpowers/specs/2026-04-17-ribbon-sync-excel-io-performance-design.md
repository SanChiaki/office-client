# OfficeAgent Ribbon Sync Excel 读写性能优化设计说明

日期：2026-04-17

状态：设计已确认，待进入实施计划

## 1. 目标

在不改变当前 Ribbon Sync 业务行为和元数据模型的前提下，显著降低以下两类操作的 Excel 端耗时：

- 全量下载时，将服务端数据刷新到当前工作表
- 全量上传时，从当前工作表读取所有待上传数据

本次设计同时覆盖以下次级目标：

- 降低部分下载 / 部分上传中的重复读写开销
- 保持当前 `_Settings`、表头识别、选区解析和连接器协议不变
- 保持“受管列按当前表头文本重新识别”的现有行为不变

## 2. 当前问题

当前性能瓶颈主要来自 Excel COM 往返调用过多。

### 2.1 全量下载写入过细

当前全量下载在 `WorksheetSyncExecutionService.WriteFullWorksheet()` 中按“行 x 列”双重循环逐个单元格调用 `gridAdapter.SetCellText()`。

这意味着：

- 每个数据单元格都是一次 COM 写入
- 新建表头时，表头文本也是逐单元格写入
- 双层表头还会逐块调用 `Merge()`

当工作表行数和列数上来后，写入时间会明显增长。

### 2.2 全量上传读取过细

当前全量上传在 `WorksheetSyncExecutionService.ReadAllCurrentCells()` 中：

- 先逐行读取 ID 列
- 再逐列读取每个业务单元格
- 每个读取动作都通过 `gridAdapter.GetCellText()` 走一次 COM 访问

这意味着总读取次数接近：

- 行数 x (1 个 ID 列 + 所有业务列)

此外，当前读取的是 Excel 单元格的 `Text`，而不是原始值，Excel 还需要参与格式化后的显示值计算，进一步放大读取成本。

### 2.3 部分上传 / 部分下载存在重复访问

当前部分同步虽然数据量较小，但仍然存在：

- 同一行被选中多个单元格时，重复回读行 ID
- 选区读取逐格访问 `Range.Cells`
- 部分下载逐目标单元格写回

这不是最大的热点，但在大面积非连续选区下也会产生明显额外开销。

## 3. 本次设计范围

### 3.1 本次要做

- 为 Excel 工作表网格适配层增加批量读写能力
- 将全量下载的数据区写入改为批量写
- 将全量上传的数据区读取改为批量读
- 优化部分上传 / 部分下载中的重复 ID 读取
- 为重型 Excel 操作增加统一的执行作用域，临时关闭界面刷新和事件

### 3.2 本次不做

- 不改变上传 / 下载接口协议
- 不引入增量上传
- 不引入快照表或脏数据追踪
- 不改变 `_Settings` 的元数据结构
- 不尝试解决 Excel `Ctrl+Z` 无法撤销插件写入的问题
- 不对整套同步引擎做大规模重构

## 4. 方案对比

### 4.1 方案一：热点批量化，局部保守回退

核心思路：

- 在现有 `IWorksheetGridAdapter` 上补充批量读写接口
- 全量下载改为按连续受管列段批量写入
- 全量上传改为按受管区域批量读取
- 部分同步继续走稀疏单元格路径，但增加缓存避免重复读取

优点：

- 改动集中，风险可控
- 对当前代码结构侵入较小
- 性能收益最直接

缺点：

- 需要处理 `Text` 与 `Value2` 的语义差异
- 下载写入必须处理“受管列不连续”的情况

### 4.2 方案二：引入统一工作表快照层

核心思路：

- 先把当前工作表受管区域一次性读成内存快照
- 上传、下载、选区解析都基于快照工作

优点：

- 长期扩展性更好
- 后续如果加脏数据分析更容易

缺点：

- 这轮范围过大
- 需要重排较多现有调用链

### 4.3 方案三：重做同步引擎为块级执行模型

核心思路：

- 所有表头、数据、清空、写入都抽象为矩形块
- 统一交由批量执行器处理

优点：

- 理论上最整洁

缺点：

- 对当前项目阶段不现实
- 风险和回归面都过大

### 4.4 结论

本次采用方案一。

原因：

- 它能解决当前最明显的真实瓶颈
- 它不要求重构既有 Ribbon Sync 业务模型
- 它能在保持用户可见行为稳定的情况下提供最大性价比

## 5. 推荐设计

## 5.1 扩展网格适配层

在当前 `IWorksheetGridAdapter` 基础上新增批量接口，供同步执行服务使用。

建议新增的语义能力：

- 读取指定矩形区域的二维值矩阵
- 将二维值矩阵批量写入指定左上角位置
- 保留现有单单元格读写接口，供稀疏选区等低密度场景继续使用

设计原则：

- Ribbon Sync 业务层不直接依赖 Excel Interop `Range.Value2`
- 仍由适配层负责屏蔽 Excel COM 细节

## 5.2 全量下载改为按连续受管列段批量写

全量下载不能直接对“第 1 列到最后一列”一次性整块覆盖写入。

原因：

- 当前工作表允许存在用户自定义列
- 受管列在运行时可能不是连续的
- 如果做整块覆盖，会误伤受管列之间的非受管列

因此本次设计要求：

- 先根据 `RuntimeColumns.ColumnIndex` 识别出所有连续受管列段
- 每个连续列段单独生成一个二维数组
- 每个连续列段分别执行：
  - 数据区清空
  - 数据区批量写入

表头处理规则：

- 如果当前使用已有布局，则只处理数据区，不重写表头
- 如果当前需要初始化表头，表头仍按现有 `WorksheetSchemaLayoutService` 结果单元格级写入和合并
- 表头区与数据区执行路径保持分离，避免 merged cell 对批量数据写入造成影响

## 5.3 全量上传改为受管区域批量读取

全量上传读取时：

- 根据当前匹配到的受管列，确定最小受管列覆盖范围
- 从 `DataStartRow` 到 `LastUsedRow` 一次性批量读出受管区域矩阵
- 在内存中解析每行的 ID 和业务列值
- 构造 `CellChange[]`

这里需要增加一个值归一化步骤：

- 将 Excel 返回的原始值统一转换为现有上传链路可接受的字符串表示
- 保持空值、数字、日期、布尔值等场景行为可预测

本次不要求保留“完全等于当前 `Text` 的显示格式”，但必须为受影响类型提供明确规则和测试覆盖。

## 5.4 部分上传 / 部分下载做轻量优化

部分同步仍然保留稀疏单元格路径，不引入大面积矩阵读写。

本次仅做以下优化：

- 在一次部分同步操作内缓存 `row -> rowId`
- 避免同一行有多个目标单元格时重复读取 ID
- 保留现有“仅可见单元格优先”和“支持非连续选区”的行为

原因：

- 部分同步通常是稀疏访问
- 为它单独构建复杂批量块收益有限
- 用缓存降低重复 COM 访问已经足够

## 5.5 增加 Excel 重型操作作用域

在全量下载、初始化大表头、全量上传批量读取这类重操作期间，增加统一的 Excel 执行作用域。

作用域内临时控制：

- `ScreenUpdating = false`
- `EnableEvents = false`
- 必要时临时关闭自动计算

并要求：

- 必须使用统一封装
- 必须在 `try/finally` 中恢复 Excel 状态
- 业务逻辑层不得散落编写状态切换代码

## 6. 架构边界

### 6.1 保持不变的边界

- `WorksheetSyncService` 对外接口不变
- 连接器协议不变
- `_Settings` 结构不变
- 表头识别逻辑和映射语义不变

### 6.2 本次新增的边界

新增一个清晰的职责边界：

- `WorksheetSyncExecutionService`
  - 负责编排本次同步要读 / 写哪些区域
- `IWorksheetGridAdapter`
  - 负责把这些区域转换为 Excel 批量读写操作
- 值归一化器
  - 负责把 Excel 原始返回值转换为上传链路的字符串值
- Excel 操作作用域
  - 负责在一次重型操作期间安全管理 Excel 全局状态

## 7. 风险点

### 7.1 上传读取语义变化

这是本次最大风险。

当前上传读取依赖 `cell.Text`，拿到的是用户看到的显示文本；批量读取通常会拿到原始 `Value2`。

可能带来的问题：

- 日期被读成 Excel 序列值
- 百分比、货币等格式变为原始数字
- 长数字或编码文本展示形式变化

缓解要求：

- 在实现前明确值归一化规则
- 用测试覆盖日期、数字、文本、空值、布尔值
- 如某类值无法安全归一化，可对该类值保留兼容回退路径

### 7.2 受管列不连续导致误覆盖

如果全量下载对大矩形整块写入，而受管列之间存在用户自定义列，则会误覆盖这些列。

缓解要求：

- 必须按连续受管列段拆分批量写
- 禁止直接对“从最小列到最大列”的完整矩形做盲写

### 7.3 双层表头和合并单元格冲突

双层表头存在合并单元格。若表头区和数据区处理不分离，批量写可能触发表头区域冲突或覆盖错误。

缓解要求：

- 表头初始化和数据区写入必须分阶段执行
- 数据批量写不得覆盖表头区

### 7.4 Excel 状态未恢复

如果临时关闭界面刷新、事件或计算后发生异常，Excel 可能残留异常状态。

缓解要求：

- 必须统一封装作用域
- 严禁在业务代码中散落手写状态切换

### 7.5 大表内存占用

批量读取 / 写入会引入 `object[,]` 矩阵。

风险：

- 行数和列数极大时，瞬时内存占用增加

缓解要求：

- 仅处理受管区域，不读取整张 sheet
- 如受管区域过大，允许按固定行块分批处理

## 8. 分阶段实施建议

建议按以下顺序实施：

1. 扩展 `IWorksheetGridAdapter` 批量读写能力
2. 将全量下载数据区改为按连续列段批量写
3. 将全量上传读取改为批量读 + 值归一化
4. 为部分上传 / 部分下载增加行 ID 缓存
5. 加入 Excel 重型操作作用域

这样做的原因是：

- 每一步都有独立收益
- 每一步都能单独回归验证
- 能把最高风险的“上传值语义变化”隔离在单独阶段内处理

## 9. 测试策略

本次实现需要覆盖以下测试层次。

### 9.1 单元测试

- 批量读取的矩形边界计算
- 连续受管列段拆分
- Excel 原始值到上传字符串的归一化规则
- 行 ID 缓存行为

### 9.2 现有执行链路回归测试

- 全量下载仍能在已有布局下只刷新数据区
- 全量下载仍能在空表头时正确生成表头和数据
- 全量上传仍按 `DataStartRow` 和运行时匹配列生成 `CellChange`
- 部分上传 / 部分下载仍支持非连续可见选区

### 9.3 手工验证

- 大表下载耗时明显下降
- 大表全量上传预览生成耗时明显下降
- 双层表头场景不被破坏
- 用户自定义插入列不被全量下载误覆盖
- 日期 / 数字 / 文本上传值符合预期

## 10. 非目标与后续议题

以下议题不是本次性能优化的目标，但后续可以继续讨论：

- 全量上传是否要做真正的脏数据筛选
- 插件写入后如何支持更好的撤销体验
- 是否把部分同步也升级为块级执行
- 是否进一步抽象统一的工作表快照层
